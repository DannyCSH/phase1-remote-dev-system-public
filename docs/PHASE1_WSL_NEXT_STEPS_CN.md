# Phase 1 的 WSL 后续路线

## 当前结论

这套公开版不把 `WSL + tmux` 当成必需前置。

也就是说：

- Windows 原生链路应该先能独立跑通
- WSL 属于后续增强，不是当前启动门槛

## 为什么先不强绑 WSL

原因通常不是“项目逻辑必须依赖 WSL”，而是：

- 某些机器没有管理员权限
- 某些机器的 Windows 版本过旧
- 某些机器还没启用 `wsl.exe`

因此更稳妥的发布方式是：

1. 先把 Windows 原生版本跑通
2. 再根据机器条件追加 WSL

## 如果你后续要升级到 WSL

建议顺序：

1. 用管理员身份打开 PowerShell
2. 启用 Windows 的 WSL 相关功能
3. 重启系统
4. 安装一个 Linux 发行版
5. 再考虑把长任务运行环境迁到 `WSL + tmux`

## 适合什么时候再做

当你已经满足以下条件时，再做会更顺：

- 当前 Windows 原生 Phase 1 已稳定可用
- 你确实需要更长驻的终端会话
- 你可以接受额外的系统配置和排障成本
